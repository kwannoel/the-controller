# Architecture Explorer Design

## Feeling

Reading a repo's architecture should feel like standing over a clean systems map: oriented immediately, never buried in implementation noise.

## Research

### CodeSee Maps

Reference: https://docs.codesee.io/docs/explore-your-map

What works:
- The map stays high-level first and lets a secondary surface carry the detailed inspection work.
- Dependencies remain visible even when files and folders are not all expanded, which preserves the system shape.
- The file browser gives users a second, more stable navigation surface than the diagram alone.

What does not fit here:
- The full interactive canvas, hide/show controls, and manual layout editing are too much UI for a first architecture pass inside The Controller.
- A folder tree duplicates the app's existing sidebar and would pull attention away from the architecture summary.

Lesson taken:
- Keep the diagram visual and high-level, but pair it with a more structured inspection rail so users do not need to navigate the graph directly for every task.

### Sourcegraph Code Navigation

Reference: https://sourcegraph.com/docs/code-search/code-navigation

What works:
- It treats structural precision and semantic understanding as complementary rather than interchangeable.
- It helps users understand dependencies and move through unfamiliar code without requiring a fully custom visualization for every workflow.
- It degrades from richer intelligence to lighter heuristics when deeper indexing is unavailable.

What does not fit here:
- Symbol-level navigation and deep code-intel affordances are too fine-grained for this feature's first version.
- Cross-repository and language-server-style precision would expand scope far beyond a high-level architecture view.

Lesson taken:
- Use a hybrid input pipeline: deterministic repo evidence for consistency, then a single AI pass for semantic grouping, naming, and summaries.

### Mermaid

Reference: https://mermaid.js.org/config/usage

What works:
- Text-based diagrams are easy to generate from structured output and easy to re-render.
- Mermaid gives a straightforward rendering target without building a custom graph engine.
- Its node identifiers can double as stable ids for syncing the diagram to the component list.

What does not fit here:
- Mermaid is not a full interaction framework; rich graph traversal, drag layout, and custom edge behaviors would become brittle quickly.

Lesson taken:
- Treat Mermaid as the visual projection of the architecture pass, not as the primary state container for interaction logic.

## Design

### Layout

Add a new `Architecture` workspace mode. The main area splits into two zones:

- Left: a large architecture canvas that renders a generated Mermaid diagram inside a scrollable viewport.
- Right: an inspection rail with a component list at the top and a details pane below it.

The left side is for orientation. The right side is for comprehension. Users should be able to keep the architecture visible while reading summaries and dependencies without opening another modal or replacing the main view.

### Hierarchy

The eye should land on the generated diagram first, then the selected component summary, then the relationship and evidence details.

Information order in the right rail:
- Component list
- Selected component summary
- Outgoing relationships
- Incoming relationships
- Evidence paths and representative snippets

The component list is not a duplicate of the sidebar. It is the semantic index generated by the same architecture pass that produced the Mermaid graph.

### Generation Pipeline

`Generate Architecture` runs against the focused project's `repo_path`.

The backend performs a bounded repo scan before invoking AI:
- top-level directories and notable config files
- dependency or import edges that can be inferred cheaply
- representative entry points and high-signal files
- README and package metadata when present
- short snippets from selected files rather than full-file dumps

That evidence is passed to a single AI architecture prompt. The model returns one JSON payload containing:
- `title`
- `mermaid`
- `components`

Each component contains:
- `id`
- `name`
- `summary`
- `contains`
- `incoming_relationships`
- `outgoing_relationships`
- `evidence_paths`
- `evidence_snippets`

The prompt requires every Mermaid node id to match a component `id` exactly so the frontend can synchronize list selection with SVG node focus.

### Interactions

Primary flow:
- User switches to `Architecture` workspace mode.
- If no architecture exists yet for the focused project, show an empty state with a single `Generate Architecture` action.
- While generating, show a loading state in both the diagram area and the right rail.
- When generation succeeds, render the Mermaid diagram and populate the component list.

Selection behavior:
- Clicking a component in the list sets `selectedComponentId`.
- The matching Mermaid node is highlighted and scrolled into view.
- The details pane updates with summary, relationships, and evidence.
- Clicking a supported Mermaid node label updates the same `selectedComponentId`.

Project switching behavior:
- Architecture is scoped per project.
- Switching focus to a different project while in `Architecture` mode swaps to that project's cached architecture view if available; otherwise it shows the empty state for that repo.

### Visual Treatment

Stay inside the app's Catppuccin Mocha palette and keyboard-first feel.

Visual decisions:
- The diagram viewport uses the same dark surface family as the rest of the workspace, but with slightly lighter panel framing so the graph reads as a map rather than another editor.
- The selected component should be obvious through one strong accent treatment only: outline and label emphasis, not multiple competing badges.
- The component list stays dense and scannable, with one-line names and subdued secondary metadata.
- The details pane should read like a system briefing: short summary first, then compact relationship sections, then evidence paths in monospace.

### States

Empty:
- No architecture generated yet for this project.
- Explain that the view is AI-generated from code structure and representative evidence.

Loading:
- Show non-blocking generation progress with a simple busy state.
- Keep the previous result visible when re-generating, but dim it and mark it as refreshing.

Error:
- Distinguish scan failure, AI command failure, invalid AI payload, and Mermaid render failure when possible.
- Never render raw invalid model output directly in the UI.

Populated:
- Default-select the first component returned by the architecture pass.
- Show a subtle note that the architecture is a high-level generated view, not an exhaustive code index.

Large repos:
- The backend should truncate evidence deliberately and tell the model that omitted files are expected.
- The UI should not imply completeness beyond the generated high-level model.

## Critique

**Eye movement**
- The original idea centered on traversing the diagram itself.
- That put too much cognitive weight on the canvas.
- The revised layout moves comprehension to the right rail so the eye can stabilize on one selected component after first taking in the overall shape.

**Negative space**
- A three-column layout with both a file tree and a component list would have overfilled the workspace.
- The file-tree concept was removed because the existing sidebar already handles project navigation.

**Visual weight**
- The details pane belongs under the list rather than as a floating overlay.
- An overlay would compete with the diagram and fragment the reading order.

**Contrast as communication**
- One selected accent state is enough.
- Extra badges, colored relationship chips, or multi-tone node states were removed because they would turn the map into a legend-decoding exercise.

**Edge cases as design inputs**
- Re-generation, invalid AI payloads, and large repos all need explicit states.
- Without those states, the feature would look trustworthy right up until it fails.

**Motion as meaning**
- Only selection sync should animate, and only enough to scroll the matching node into view.
- Zoom choreography and node-to-node flyovers were removed because they add motion without meaning.

**Density vs. cognitive load**
- The component list remains semantic and high-level, not a raw folder listing.
- The diagram should summarize structure; the right rail should summarize meaning.

**The glance test**
- In half a second, a user should understand three things: which repo they are looking at, the broad architecture shape, and which component is currently selected.

**Consistency as trust**
- Making `Architecture` a workspace mode instead of a one-off modal keeps the feature aligned with `Development`, `Agents`, and `Notes`.

Removed or simplified after critique:
- direct graph-traversal-first interaction
- separate folder/component browser inside the architecture view
- freeform diagram dragging and editing
- multi-pane overlays for component details
